# Agent Governance Kit — Installation and Customization Instructions

> **Read this file FIRST.** This document tells an AI agent (or human) how to install and configure the Agent Governance Kit for any software project.

---

## What This Kit Is

This is a portable, project-agnostic **AI agent governance framework**. It provides:

1. **System instructions** (`AGENTS.md` / `CLAUDE.md`) — The first file any AI agent reads. Defines the operating model, constraints, workflows, and process architecture.
2. **Plan template** (`PLAN_TEMPLATE.md` + schemas) — A formal, structured template for execution plans that ensures consistent, traceable, deterministic agent work.
3. **Prompt template** (`master-maths-prompt-template.md`) — A reusable prompt template using the MATHS structure for governed work. It is separate from {{PROJECT_NAME}} (MA).

Together, these files create a **deterministic execution framework** that bridges the gap between probabilistic LLM behavior and the deterministic requirements of production software.

---

## Why This Framework Exists

### The Problem

LLMs are probabilistic. When you ask an AI agent to implement a feature, it makes educated guesses at every step. 90% accuracy per step sounds good — until you realize that over 5 steps, that is approximately 59% end-to-end accuracy. Over 10 steps: approximately 35%.

Without structure, AI agents:
- Drift from the original task scope
- Invent information they do not have
- Skip validation steps
- Make changes without approval
- Forget to document what they did
- Produce inconsistent results across runs

### The Solution

This framework pushes **reliability** into deterministic processes (plans, templates, validation gates) and reserves **flexibility** for the LLM's reasoning. The agent operates within strict constraints:

- **Plans** define WHAT to achieve (formal, structured, traceable)
- **Prompts** define HOW to execute (phased, gated, deterministic)
- **Rules** define WHAT IS FORBIDDEN (hard blocks, no workarounds)
- **Governance docs** define WHY (vision, goals, success metrics)
- **Tools** execute work DETERMINISTICALLY (scripts, validators, linters)
- **The agent** ORCHESTRATES (reads plans, applies rules, uses tools, reports results)

## What Is the {{PROJECT_NAME}} (MA) Process

> **Authoritative standard:** For the complete, definitive explanation of {{PROJECT_NAME}} — including theory, rationale, operator theory, falsification, governance layering, and team topology — read `agent_governance/mathematical_autopsy:_deterministic_ai_creation.md`. The summary below introduces the key concepts; the whitepaper is the tiebreaker on any ambiguity.

The **{{PROJECT_NAME}} (MA)** is a rigorous, phased process for implementing any code that involves mathematical operations, algorithms, or performance guarantees. It prevents the common failure mode where math code is written, "seems to work," but has subtle bugs, non-determinism, or undocumented assumptions that surface later.

### Why MA Exists

Math and algorithm code has unique risks:
- **Non-determinism**: Floating-point arithmetic, sort stability, and unseeded randomness can produce different results across runs.
- **Silent failure**: A scoring function can return plausible-looking but incorrect results.
- **Undocumented assumptions**: If the developer (or AI) assumes inputs are always positive but does not check, a negative input silently corrupts results.
- **Untestable claims**: "This algorithm is O(n log n)" is meaningless without proof.

### How MA Works

MA is a separate methodology from the `MATHS` prompt template. When MA applies, follow the MA phases and deliverables defined by the active repository rules and MA documentation directly. Do not infer or substitute MA phases from prompt-template letters.

### Key MA Concepts

- **Invariant**: A machine-enforced guarantee defined in a YAML file. Example: "Scoring produces bit-identical outputs given identical inputs and seeds." Invariants are tested automatically and block deployment if violated.
- **Lemma**: A formal mathematical claim with a proof sketch. Example: "The composite score is strictly monotone in each sub-score given positive weights." Lemmas provide the mathematical reasoning that invariants enforce.
- **Scorecard**: A JSON artifact generated by running a verification notebook. Contains pass/fail results for every invariant, seed-lock verification, and a GO/NO-GO decision.
- **Notebook-first development**: All math/algorithm code is written and tested in Jupyter notebooks first, then extracted to Python files only after the scorecard is green. This ensures code is validated before it enters the codebase.
- **Fail-closed**: When an invariant is violated (e.g., NaN detected in a score), the system rejects the result rather than silently proceeding. This is a core safety principle.

### When MA Applies

MA is **required** for:
- New mathematical operations or algorithms
- New performance guarantees (latency, throughput, accuracy thresholds)
- Changes to existing math that affect correctness
- Any code involving computation with determinism requirements

MA is **not required** for:
- Trivial bug fixes that do not change mathematical behavior
- Documentation-only changes
- Configuration changes that do not affect algorithms
- Pure infrastructure (CI pipelines, deployment scripts)

## What Is the MATHS Framework

MATHS is an acronym for **Model, Annotate, Tie, Harness, Stress-test**. It is a reusable prompt-template structure. It is not the MA process.

### The Five Phases

1. **M — Model**: Define the problem, stakeholders, success metrics, constraints, and out-of-scope. This is the "what and why."

2. **A — Annotate**: Define the formal rules, assumptions, and data contracts. This is the "logic and structure."

3. **T — Tie**: Define dependencies, failure modes, and the go/no-go decision. This is the "integration and decision framing."

4. **H — Harness**: Create the implementation plan and execution harness. This is the "build and verify" section of the prompt.

5. **S — Stress-test**: Define validation, stress checks, and acceptance gates. This is the "validate and certify" section of the prompt.

If a task separately requires MA, list those MA deliverables explicitly in the prompt or plan. That is a task-level choice, not a built-in property of MATHS.

### Phase Transition Protocol

Each phase must be completed before the next begins. After completing a phase, the agent emits an exit checklist and waits for confirmation. If blocked, the agent emits a BLOCKED signal and does not proceed.

## Installation Steps

### Step 1: Place System Instructions at Project Root

Copy these two files from `agent_governance/` to the **project root**:

```
agent_governance/AGENTS.md  →  <project-root>/AGENTS.md
agent_governance/CLAUDE.md  →  <project-root>/CLAUDE.md
```

These files are identical except for the title line (AGENTS.md is for general agents, CLAUDE.md is for Claude-family models). Both must exist at the project root.

### Step 2: Keep Templates in agent_governance/

The following files remain in `agent_governance/` (do not move them):

```
agent_governance/PLAN_TEMPLATE.md
agent_governance/PLAN_TEMPLATE_SCHEMA.yaml
agent_governance/PLAN_TEMPLATE_SCHEMA.json
agent_governance/master-maths-prompt-template.md
```

### Step 4: Create Governance Documents

Create these files with project-specific content. These are the documents that AGENTS.md/CLAUDE.md reference.

#### 4a: North Star Document (`docs/governance/NORTH_STAR.md`)

This is your project's vision document. It defines:
- Mission statement
- Vision and goals
- Core success metrics (measurable)
- Technical excellence goals
- Key principles that all work must align with

Every piece of work an AI agent does must align with this document. If it does not align, the agent must stop and ask.

#### 4b: Execution Plan (`docs/governance/EXECUTION_PLAN.md`)

This defines what work is authorized. Structure it as a phased plan with:
- Phases or milestones
- Tasks within each phase (with checkboxes)
- Status indicators (Not Started, In Progress, Complete)
- Dependencies between tasks

AI agents can only work on tasks that appear in this document (or in detailed plans in `/plans/`).

#### 4c: Action Log (`docs/governance/ACTION_LOG`)

Start this as an empty file. AI agents append entries after every action:

```
YYYY-MM-DD HH:MM:SS TZ — Brief description: detailed explanation with plan reference and North Star alignment.
```

Newest entries go at the top.

#### 4d: Project Status (`docs/governance/PROJECT_STATUS.md`)

High-level project status including:
- Current phase
- Overall progress
- Active blockers
- Recent milestones

### Step 5: Customize All Files for Your Project

Every `<FILL:...>` placeholder in the template files must be replaced with project-specific values.

#### 5a: Customize AGENTS.md and CLAUDE.md

Open both files and update:

1. **Title and description** — Replace `<FILL:project-name>` with your project name.
2. **North Star key metrics** — Replace the placeholder metrics with your project's actual key metrics from `docs/governance/NORTH_STAR.md`.
3. **File Structure Map** — Update the directory table to match your project's actual structure. Add project-specific directories, remove ones that do not exist.
4. **Operating Model table** — Update paths if your project uses different directory conventions.
5. **Rules listing** — If your project has `.cursor/rules/` files, list them. If not, note that rules will be created as the project matures.
6. **Codex-specific section** — Remove or customize if you do not use Codex agents.
7. **Python environment** — Confirm `.venv/bin/python` is correct, or update to your project's convention.

#### 5b: Customize PLAN_TEMPLATE.md

Open and update:

1. **Governance lock references** — Update file paths to match your project structure.
2. **Schema reference paths** — Confirm `agent_governance/PLAN_TEMPLATE_SCHEMA.yaml` / `.json` paths are correct.
3. **Inline examples** — Replace the generic examples with examples relevant to your project domain. Keep the structure identical; only change the content.
4. **"How to use" section** — Update the copy paths if your plan storage convention differs.

#### 5c: Customize master-maths-prompt-template.md

Open and update:

1. **Governance lock references** — Update file paths.
2. **Canonical vocabulary** — Add project-specific terms when the plan or vocabulary table requires them; define each explicitly on first use.
3. **Examples** — Replace or augment the generic examples with your domain.
4. **Codex references** — Remove if you do not use Codex.

#### 5d: Customize Schemas (YAML and JSON)

Open and update:

1. **`north_star_ref`** — Update the path to your North Star document.
2. **`governance_closure` paths** — Update action log and execution plan paths.
3. **Title and description** — Replace generic names with your project name.

### Step 6: Create Invariant Template (if MA applies)

If your project involves math or algorithms, create `invariants/INV_TEMPLATE.yaml`:

```yaml
id: INV-XXXX
name: <descriptive-name>
owner: <team-or-individual>
status: proposed
description: >
  Clear description of what this invariant guarantees.
equations:
  - "<equation 1>"
thresholds:
  <threshold_name>: <value>
acceptance_checks:
  - <artifact>.json.<path> == <value>
producer: notebooks/math/<notebook_name>.ipynb
artifacts:
  - configs/generated/<artifact_name>.json
rollback_criteria: >
  When should we rollback?
```

And create `invariants/INDEX.yaml`:

```yaml
invariants: []
```

### Step 7: Verify Installation

After completing all steps, verify:

- [ ] `AGENTS.md` exists at project root with project-specific content
- [ ] `CLAUDE.md` exists at project root (identical to AGENTS.md except title)
- [ ] `agent_governance/` directory contains all template files
- [ ] `docs/governance/NORTH_STAR.md` exists with project vision
- [ ] `docs/governance/EXECUTION_PLAN.md` exists with authorized work
- [ ] `docs/governance/ACTION_LOG` exists (may be empty)
- [ ] `docs/governance/PROJECT_STATUS.md` exists
- [ ] `plans/` directory exists
- [ ] `docs/prompts/` directory exists
- [ ] All `<FILL:...>` placeholders in all files have been replaced

## Process Architecture Summary

```
User Request
    ↓
Agent reads AGENTS.md / CLAUDE.md (system instructions)
    ↓
Agent reads North Star (docs/governance/NORTH_STAR.md)
    ↓
Agent reads Execution Plan (docs/governance/EXECUTION_PLAN.md)
    ↓
Plan exists for this work?
  ├── YES → Follow plan, get change approval from user
  └── NO  → Create plan using agent_governance/PLAN_TEMPLATE.md, get approval
    ↓
Math/Algorithm scope?
  ├── YES → Execute the required methodology directly (MA when required by scope/plan)
  │         Prompt-template selection, including MATHS, is independent
  └── NO  → Execute per plan with standard change approval
    ↓
Implementation
    ↓
Validation (tests, lints, invariant checks, scorecard)
    ↓
Post-Work (action log entry, execution plan update, governance closure)
```

## Key Principles for Agents

1. **Read before writing.** Always read a file before modifying it.
2. **Plan before executing.** Every unit of work needs a plan.
3. **Ask before acting.** Get explicit user approval ("go") before any write operation.
4. **Tool before guessing.** Use deterministic tools; never guess what a tool can tell you.
5. **Fix before working around.** Diagnose and fix root causes; never use workarounds.
6. **Document after doing.** Update the action log after every action.
7. **Escalate before inventing.** When stuck, tell the user. Never fabricate information.
8. **Validate after changing.** Run tests and lints after every modification.
